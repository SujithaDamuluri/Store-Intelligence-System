# CHOICES.md

## Decision 1: Detection Model Selection

### Problem

The system requires accurate customer detection from CCTV footage under varying lighting conditions and store layouts.

### Options Considered

1. YOLOv8
2. YOLOv9
3. RT-DETR
4. Faster R-CNN

### AI Suggestion

AI-assisted evaluation suggested YOLOv8 due to:

* Fast inference speed
* High accuracy
* Easy integration with OpenCV
* Strong documentation

### Final Choice

YOLOv8

### Why

YOLOv8 provides a strong balance between speed and accuracy. It was simple to integrate into the pipeline and performed reliably on retail CCTV footage.

### Trade-offs

Pros:

* Fast
* Lightweight
* Easy deployment

Cons:

* May struggle under severe occlusion
* Requires confidence threshold tuning

## Decision 2: Tracking Strategy

### Problem

The challenge requires assigning persistent visitor identities across frames.

### Options Considered

1. ByteTrack
2. DeepSORT
3. StrongSORT
4. Custom Tracking

### AI Suggestion

AI-assisted comparison recommended ByteTrack because:

* Simpler setup
* Strong real-world performance
* Good compatibility with YOLO

### Final Choice

ByteTrack

### Why

ByteTrack provided stable tracking IDs while requiring minimal additional configuration.

### Trade-offs

Pros:

* Easy integration
* Fast
* Accurate for moderate crowd density

Cons:

* Can lose identity under heavy occlusion
* No advanced appearance-based re-identification

## Decision 3: Backend Framework

### Problem

The analytics layer requires REST APIs, validation, and documentation.

### Options Considered

1. Flask
2. Django
3. FastAPI

### AI Suggestion

AI-assisted analysis suggested FastAPI because:

* Automatic Swagger documentation
* Pydantic validation
* High performance

### Final Choice

FastAPI

### Why

FastAPI accelerated development and provided automatic API documentation through Swagger UI.

### Trade-offs

Pros:

* Fast development
* Built-in validation
* Modern async support

Cons:

* Smaller ecosystem than Django

---

## Decision 4: Event Storage Strategy

### Problem

Events generated by the detection pipeline need to be stored and processed.

### Options Considered

1. JSONL files
2. SQLite
3. PostgreSQL
4. Kafka Event Stream

### AI Suggestion

For a challenge implementation, JSONL storage was recommended due to simplicity.

### Final Choice

JSONL

### Why

JSONL allowed rapid development and easy inspection of generated events without requiring additional infrastructure.

### Future Upgrade

Move to PostgreSQL and Kafka for production deployment.
